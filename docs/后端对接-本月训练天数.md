## 4. `monthTrainedDays` 精确口径（务必和前端一致）

| 项 | 定义 |
|---|---|
| 统计范围 | **当前自然月**（按服务器「当前北京时间」的年-月） |
| 时区 | **北京时间 UTC+8**（与其它 `/stats/*` 接口一致） |
| 取值字段 | `workout_sessions.started_at`（按它落在哪个北京日） |
| 计入条件 | `status = 'completed'` 且 `deleted_at IS NULL` |
| 去重 | 按**北京日期**去重，同一天多次训练只算 1 天 |
| 有效门槛 | **无**——任何 completed 训练都算 1 天（不要求 ≥2 动作 / ≥6 组） |
| 空数据 | 返回 `0` |

**SQL 思路（PostgreSQL）：**

```sql
SELECT COUNT(DISTINCT (started_at AT TIME ZONE 'Asia/Shanghai')::date) AS month_trained_days
FROM workout_sessions
WHERE user_id = :uid
  AND status = 'completed'
  AND deleted_at IS NULL
  AND (started_at AT TIME ZONE 'Asia/Shanghai')
        >= date_trunc('month', (now() AT TIME ZONE 'Asia/Shanghai'))
  AND (started_at AT TIME ZONE 'Asia/Shanghai')
        <  date_trunc('month', (now() AT TIME ZONE 'Asia/Shanghai')) + interval '1 month';
```

> ⚠️ 时区一致性：前端目前用设备本地时间计算（国内用户即 UTC+8，与上式一致）；待前端切到聚合接口后，以后端北京时区口径为准，月初/月末边界不会再有歧义。

## 5. 验收

- `GET /api/v1/stats/home` 返回里有 `monthTrainedDays`（integer，≥0），无 `activeChallenge`。
- 同一天练 2 次，`monthTrainedDays` 只 +1。
- 跨月后旧月份的训练不再计入（自然月归零重算）。
- `api.json` 已移除/标记 challenges 接口与 `ChallengePlan`/`UserChallenge`。

- 不需要保留 challenges 的进度写入逻辑（`completedSessions` 自增等）。
- 本次不涉及热力图 / 趋势 / PR 等其它 `/stats/*` 接口改动（那些前端切换是另一轮的事）。
