# 后端对接 · 训练空闲自动结束 + 待领奖励

> 解决「用户不点完成、训练时长一直累加（曾出现 230 小时）」的问题。
> 字段级以 `KaiLift/docs/api/openapi.json`（或 `GET /api/docs/ui`）为单一事实源，本文是前端落地指引。

## 一、规则（与产品确认）

从**最后一组**算起的空闲时长触发，有新组就重置：

| 空闲时长 | 动作 |
|---|---|
| 30 分钟 | 弹提醒一（轻提示） |
| 45 分钟 | 弹提醒二（更明确） |
| 60 分钟 | **自动结束本次训练** |

- **结束时间钳位**：后端统一把 `finishedAt = min(触发时刻, 最后一组时间 + 3min)`，所以时长不会再爆表（哪怕隔天补点完成也被钳到末组+3min）。前端不用自己算时长。
- **空组**训练（一组没记）空闲超时 → 后端自动 `cancelled`，不计入统计/成就。
- **自动结束的奖励（升级/升段/新解锁）不当场弹**，存为「待领奖励」，等用户**下次进 App** 补放庆祝。

## 二、责任分工（重要）

| 能力 | 谁做 | 说明 |
|---|---|---|
| 30/45 弹窗 + 60 自动结束 | **前端**（F1–F3） | 仅在 **App 前台开着**时有效（微信切后台/锁屏会挂起 JS 定时器） |
| App 关了之后的自动结束 | **后端已做**（惰性兜底） | 用户下次开 App / 开练时，后端发现 active 空闲>60min 自动结算掉，前端**无需处理** |
| 待领奖励补庆祝 | **前端**（F4） | 进 App 时拉 `GET /rewards/pending`，有则补弹 |

> 也就是说：前端只管「用户正盯着 App」这段；离开 App 期间由后端兜底。两边阈值都是 60min，对齐即可。

## 三、接口契约（后端已上线）

### 1) 自动结束训练：复用 PATCH，加 `auto:true`

```
PATCH /api/v1/sessions/{id}    body: { "status": "completed", "auto": true }
```

- `auto:true` 时：成就/经验照常落库，但**响应不返回 `achievement` 字段**（奖励转入待领），前端**不要弹庆祝**。
- 手动结束仍是 `{ status:'completed' }`（不带 auto），响应含 `achievement`，照旧当场弹。
- 响应 `data.session` 新增字段 `autoFinished: boolean`。

### 2) 取待领奖励（进 App 时调）

```
GET /api/v1/rewards/pending
```

```jsonc
{
  "ok": true,
  "data": {
    "items": [
      {
        "sessionId": "ck…",
        "name": "推日 + 腿",
        "finishedAt": "2026-06-27T05:32:30.000Z",
        "autoFinished": true,
        "reward": {            // 结构与手动结束的 achievement 完全一致，可复用现有庆祝组件
          "expGained": 360,
          "trickleBreakdown": { "base": 100, "volume": 200, "pr": 60 },
          "level": { "level": 7, "before": 6, "isLevelUp": true, "...": "LevelProgress 同 finish" },
          "rank": { "beforeKey": "...", "isPromotion": false, "...": "RankProgress 同 finish" },
          "unlocked": [ { "key": "...", "title": "...", "icon": "...", "category": "..." } ]
        }
      }
    ]
  }
}
```

- 多条按 `finishedAt` 倒序；没有则 `items: []`。
- `reward` 与 `Session.finish` 返回的 `achievement` 同构 → **直接喂现有 `buildFinishReward` / 庆祝弹窗**。

### 3) 确认已展示，清除待领

```
POST /api/v1/rewards/ack    body: { "sessionIds": ["ck…", "ck…"] }   -> { "cleared": n }
```

- 展示成功后调，传上一步的 `sessionId`。与 GET 分离是为防「拿到后没渲染就断网」丢庆祝；**渲染完再 ack**。幂等。

### 4) 新增字段：组的 `createdAt`

`GET /api/v1/sessions/{id}` 详情里每个 `sets[]` 现在带 `createdAt`（date-time）。前端据此算「最后一组时间」。

## 四、前端落地（F1–F4）

### F0 service 层（`miniprogram/service/api.js`）

```js
// Session.finish 加 auto 参数
finish: (id, auto = false) => patch(`/api/v1/sessions/${id}`, { status: 'completed', auto }),

// 新增 Reward 服务
const Reward = {
  pending: () => get('/api/v1/rewards/pending'),            // -> { items: [...] }
  ack: (sessionIds) => post('/api/v1/rewards/ack', { sessionIds }), // -> { cleared }
};
exports.Reward = Reward;
```

### F1 维护「最后一组时间」`_lastActivityMs`（`pages/workout/workout.js`）

- 在 `_applySession(detail)`（[workout.js:214](../miniprogram/pages/workout/workout.js#L214)）里算：
  ```js
  // 末组 createdAt（无组用 startedAt）。来自服务端，切 tab 重载也稳定，不会误重置空闲钟。
  const allSets = (detail.workoutExercises || []).flatMap(w => w.sets || []);
  const lastMs = allSets.length
    ? Math.max(...allSets.map(s => new Date(s.createdAt).getTime()))
    : new Date(detail.startedAt).getTime();
  this._lastActivityMs = lastMs;
  this._idleStage = 0; // 0=未提醒 1=已弹30 2=已弹45
  ```
- **每次成功加组后**（语音/手动写库成功的回调里，加组成功 `_addSets` 之后）重置：
  ```js
  this._lastActivityMs = Date.now();
  this._idleStage = 0;
  ```

### F2 在已有 1 秒计时里判空闲（`_startTimer` 的 `tick`，[workout.js:294](../miniprogram/pages/workout/workout.js#L294)）

```js
const IDLE_REMIND_1 = 30 * 60 * 1000;
const IDLE_REMIND_2 = 45 * 60 * 1000;
const IDLE_AUTO_END = 60 * 60 * 1000;

const tick = () => {
  // …原有 seconds 更新…
  if (this.data.phase !== 'training' || !this._lastActivityMs) return;
  const idle = Date.now() - this._lastActivityMs;
  if (idle >= IDLE_AUTO_END) { this._autoFinish(); return; }
  if (idle >= IDLE_REMIND_2 && this._idleStage < 2) { this._idleStage = 2; this._idleRemind(2); }
  else if (idle >= IDLE_REMIND_1 && this._idleStage < 1) { this._idleStage = 1; this._idleRemind(1); }
};
```

```js
_idleRemind(stage) {
  wx.showModal({
    title: stage === 1 ? '还在练吗？' : '准备结束了吗？',
    content: stage === 1 ? '你已 30 分钟没有记录新动作。' : '你已 45 分钟没记录，再过 15 分钟将自动结束本次训练。',
    confirmText: '继续训练', cancelText: '结束训练',
    success: (r) => { if (!r.confirm) this._finishSession(/* 手动结束，照常弹奖励 */); },
  });
}

async _autoFinish() {
  this._clearTimer();
  try { await Session.finish(this.data.session.id, true); } catch (e) {}
  wx.showToast({ title: '已自动结束本次训练', icon: 'none' });
  this._loadActive(); // 刷新为 empty 态；奖励等下次进 App 由 F4 补放
}
```

> `_idleStage` 防重复弹；点「继续训练」不重置 `_lastActivityMs`（没真练就不该续命）——下一档到点会再弹。如果你希望「继续训练」也顺延，可在 confirm 里 `this._lastActivityMs = Date.now()`，按产品口味二选一。

### F3 `onShow` 回前台补判（[workout.js:142](../miniprogram/pages/workout/workout.js#L142)）

切后台/锁屏时定时器被挂起，回前台要补算一次。`onShow → _loadActive → _applySession` 已会重设 `_lastActivityMs`（来自服务端末组），随后下一个 tick 自然会按真实空闲触发提醒/自动结束。**额外保险**：`_loadActive` 拉到 active 后，若 `Date.now() - _lastActivityMs >= IDLE_AUTO_END` 直接 `_autoFinish()`（兜底，正常这种已被后端惰性关掉、`Session.active()` 返回 null）。

### F4 进 App 补放待领奖励

在 **App 启动**（`app.js onLaunch` 登录后）或**首页 `onShow`** 调一次：
```js
const { items } = await Reward.pending();
if (items.length) {
  for (const it of items) showFinishReward(buildFinishReward(it.reward)); // 复用现有庆祝
  await Reward.ack(items.map(it => it.sessionId));
}
```
> 放首页 onShow 最稳（每次回到首页都会补领）。多条可只弹最近一条或依次弹，按体验定。

## 五、自测清单

- [ ] 训练中静置 30/45 分钟分别弹提醒；点「继续」不消失式重弹、点「结束」走手动结束弹奖励。
- [ ] 静置 60 分钟自动结束，toast「已自动结束」，页面回到可开新训练态，**当场不弹成就**。
- [ ] 加新组后空闲钟归零（30 分提醒重新计）。
- [ ] 关掉 App 放置 >1 小时再打开/开练：旧训练已是 completed（后端兜底），不再累加时长。
- [ ] 进首页拉到待领奖励并补弹庆祝，ack 后再进不重复弹。
- [ ] 详情/分享图时长正常（≈ 末组−开始，不再出现几十上百小时）。

## 六、联调备注

- 本地联调：后端 dev server `http://192.168.1.7:20020`（`constants.LOCAL_HOST`）。
- 阈值 30/45/60 与缓冲 3min 是后端 `src/lib/api/session-lifecycle.ts` 的常量；前端 30/45/60 要与之一致（60 两边都判，缓冲只后端用）。
- 不要前端也去 PATCH 那些「已被后端兜底关掉」的 session（会 404/状态不符）——以 `Session.active()` 返回为准。
