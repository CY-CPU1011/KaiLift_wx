# 后端对接说明：训练热力图色深由「训练量/次数」改为「单日训练时长」

**日期：2026-06-20｜提出方：小程序前端｜状态：前端已改完（含降级），待后端给 `/stats/heatmap` 增加 `durationMin` + 同步 `api.json`**

---

## 1. 变更背景（一句话）

原热力图图例写的是「1–2次 / 3–5次 / 6–8次 / 9+次」，但色深其实是按当日训练量（kg）算的，文案与口径不符，而且「一天练 6~9 次」不符合现实直觉。

产品决定：**色深改为按「单日训练时长」分档**，更符合"今天练了多久"的直觉。

---

## 2. 前端现状（已完成，不依赖后端也能跑）

前端这一轮已改完并自测：

- 图例文案：`无训练 / 10分钟 / 半小时 / 1小时 / 1.5小时+`（`miniprogram/components/heatmap/index.js`）。
- 等级算法由 `volumeToLevel(kg)` 换成 `durationToLevel(分钟)`（`miniprogram/pages/stats/stats.js`），分档见 §4。
- 读取字段由 `day.volumeKg` 换成 `day.durationMin`。
- **降级逻辑**：当 `day.durationMin` 为 `null`/缺省（后端尚未返回）时，只要这天有训练（`volumeKg`/`setCount`/`sessionCount` 任一 > 0）就点亮最浅一档（level 1），避免整年空白。

> 也就是说：**后端不动，前端现在也能显示**（但只会是"有练=最浅一档"的粗略效果）。后端补上 `durationMin` 后，前端会自动切换为真正的时长分档，**无需再改前端**。

## 4. `durationMin` 精确口径（务必和前端分档一致）

### 4.1 单日时长的统计口径

| 项 | 定义 |
|---|---|
| 时区 | **北京时间 UTC+8**（与其它 `/stats/*` 接口一致） |
| 归属日期 | 按 `started_at` 落在哪个**北京日**（与同接口 `setCount`/`volumeKg` 的归日口径保持一致） |
| 计入条件 | `status = 'completed'` 且 `deleted_at IS NULL` 且 `finished_at IS NOT NULL` |
| 单次时长 | `finished_at - started_at`，换算成**分钟** |
| 单日合计 | 同一北京日的多次训练时长**求和**后取整（四舍五入到整数分钟） |
| 跨零点 | 整段时长计入 `started_at` 所在的北京日（不按零点拆分，简单且与归日口径一致） |
| 空数据 | 该日无 completed 训练则不返回该 `day`（沿用现有「只返回有训练日」的约定） |

### 4.2 前端分档（后端只需给出 `durationMin`，分档在前端做，列出供对齐验证）

| 单日 `durationMin` | level | 图例 |
|---|---|---|
| 0 或缺省 | 0 | 无训练（浅灰） |
| `0 < m < 30` | 1 | 10分钟（最浅绿） |
| `30 ≤ m < 60` | 2 | 半小时（浅绿） |
| `60 ≤ m < 90` | 3 | 1小时（绿） |
| `m ≥ 90` | 4 | 1.5小时+（深绿） |

**SQL 思路（PostgreSQL，按年聚合，可与现有按日聚合合并成一条）：**

```sql
SELECT
  (started_at AT TIME ZONE 'Asia/Shanghai')::date                       AS day,
  ROUND(SUM(EXTRACT(EPOCH FROM (finished_at - started_at)) / 60.0))::int AS duration_min
FROM workout_sessions
WHERE user_id = :uid
  AND status = 'completed'
  AND deleted_at IS NULL
  AND finished_at IS NOT NULL
  AND (started_at AT TIME ZONE 'Asia/Shanghai') >= make_date(:year, 1, 1)
  AND (started_at AT TIME ZONE 'Asia/Shanghai') <  make_date(:year + 1, 1, 1)
GROUP BY day;
```

## 5. 验收

- `GET /api/v1/stats/heatmap?year=2026` 的每个 `day` 都带 `durationMin`（integer，≥0）。
- 同一天练 2 次（如 30 分钟 + 40 分钟），该日 `durationMin = 70`、落在 level 3。
- 未结束的训练（`finished_at IS NULL`）不计入时长。
- `api.json` 的 heatmap 响应已含 `days[].durationMin`，description 与实现一致。

---

## 6. 明确「不需要」后端做的，避免误解

- 不需要在后端做分档（level 计算）——后端只给原始 `durationMin`，0~4 档由前端按 §4.2 计算（便于以后调阈值不动后端）。
- 不需要新增时长字段落库自增——实时从 `started_at`/`finished_at` 派生即可（避免改组/删组后失真）。
- 本次不动 `setCount`/`volumeKg`/`sessionCount`，它们继续返回（保留给其它视图/未来切换指标用）。
- 本次不涉及 `/stats/home`、`/stats/trend`、`/stats/day`、PR 等其它接口。
